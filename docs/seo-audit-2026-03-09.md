# SEO + Analytics Deep Audit — esdeveniments.cat

**Date**: March 9, 2026
**Data period**: Dec 6, 2025 – Mar 6, 2026 (90 days)
**Context**: v2 launched late November 2025 (Next.js 16 + i18n + Design System overhaul)

---

## Executive Summary

Since v2 launched ~3.5 months ago, the site receives **~14,800 sessions/90 days** (~165/day average) with **~7,400 from organic search** (50%). Mobile engagement is excellent (18% bounce, 73s avg duration), but desktop shows a suspicious 54% bounce rate. The site ranks for ~500+ queries but CTR is critically low across most pages (under 3%). Key structural issues include URL fragmentation (3-segment date+category URLs cannibalizing each other), missing AI referrer tracking for the site's 3rd-largest traffic source (ChatGPT), and near-zero usage of share/calendar features despite being core to an events discovery site.

## Finding 1: URL Fragmentation — 3-Segment Pages Cannibalizing Each Other

### Data

GSC shows extreme fragmentation for broad queries:
- **"actividades en santa eugènia"**: 135 impressions split across **12 different URLs** (`/es/santa-eugenia-de-berga/dema/gent-gran`, `/es/santa-eugenia-de-berga/setmana/familia-i-infants`, etc.) — all ranking pos 23-99 with **0 clicks total**
- **"actividades gratuitas en granollers"**: 49 impressions split across **16 different URLs** — all with **0 clicks**
- **"actividades en barcelona"**: 41 impressions across 3 URLs — **0 clicks**

The pattern is consistent: `/[place]/[date]/[category]` three-segment URLs create dozens of thin, competing pages per town that dilute authority.

### Why This Happens (Codebase)

The sitemap generator at `app/sitemap-places/[...parts]/route.ts` (lines 106-119) creates URLs for **every combination** of place × date × category:

```typescript
// /[place]/[date]/[category]
for (const category of topCategories) {
  const dateCatLoc = buildCanonicalUrlDynamic(
    { place: place.slug, byDate: date, category }, categories
  );
  fields.push({ loc: dateCatLoc, ... });
}
```

With `topDates` (4 values: avui, dema, setmana, cap-de-setmana) × `topCategories` (N categories), this generates 4×N URLs per place, each served to Google for indexing.

Meanwhile, the `proxy.ts` noindex logic (lines 539-549) only blocks:
- `/compartir-tiktok` and `/callback` (hardcoded paths)
- URLs with query params (`search`, `distance`, `lat`, `lon`)

**Three-segment URLs like `/es/granollers/cap-de-setmana/esport` are NOT noindexed** and ARE included in sitemaps.

### 2026 SEO Trend Alignment

Google's 2026 guidance strongly penalizes "programmatic SEO done poorly" — pages generated at scale that lack unique content (John Mueller: "sometimes a fancy banner for spam"). Each 3-segment page shows the same event list as its parent, just filtered — essentially thin/duplicate content from Google's perspective.

### Action Plan

1. **Add noindex for 3-segment URLs** in `proxy.ts`:
   - Detect paths matching `/[place]/[date]/[category]` pattern (3 segments after locale strip)
   - Add `X-Robots-Tag: noindex, follow` header
   - Keep `follow` so link equity flows to the parent place page

2. **Remove 3-segment URLs from sitemaps**:
   - In `app/sitemap-places/[...parts]/route.ts`, remove or gate the `/[place]/[date]/[category]` block (lines 106-119)
   - Keep `/[place]`, `/[place]/[date]`, and `/[place]/[category]` (2-segment URLs)

3. **Verify**: After deploying, use GSC URL Inspection to confirm 3-segment URLs return `noindex`. Monitor GSC Coverage report for "Excluded by noindex tag" increase.

## Finding 2: Desktop 54% Bounce Rate — Bot Traffic Inflation

### Data

| Device | Sessions | Engaged | Bounce | Duration |
|--------|----------|---------|--------|----------|
| Mobile | 6,498 | 5,347 (82%) | **18%** | 73s |
| Desktop | 8,271 | 3,824 (46%) | **54%** | 55s |
| Tablet | 96 | 85 (89%) | 11% | 295s |

Desktop has MORE sessions than mobile (56% vs 44%) but dramatically worse engagement. Meanwhile, GSC shows ~85% of search clicks come from mobile. This gap strongly suggests desktop sessions include significant bot/crawler traffic.

Supporting evidence:
- "(direct) / (none)" source: 6,323 sessions with 60% bounce — bots typically arrive as direct traffic
- "(not set) / (not set)": 222 sessions with **99% bounce** — definitively bot traffic
- Country data: USA (2,406 sessions), Singapore (882), China (721) with very low engagement — likely bot origins

### Why This Happens (Codebase)

The `proxy.ts` middleware doesn't perform bot detection — it focuses on HMAC security, CSP headers, and canonical redirects. There's no user-agent filtering at the middleware or analytics level.

GA4's built-in bot filtering depends on the "Known bots and spiders" checkbox in the data stream settings, but this doesn't catch custom scrapers or headless browser bots.

### 2026 SEO Trend Alignment

AI-powered scrapers (training data collection) have increased dramatically. Sites with structured data and clean HTML (like this one) are prime targets. The ChatGPT/Perplexity traffic (271 real sessions) suggests the site is already in AI training sets, which means scraper traffic will increase.

### Action Plan

1. **Verify GA4 bot filtering**: Admin → Data Streams → Web → Configure tag settings → "Define internal traffic" → verify internal IPs are excluded. Also check "Exclude all hits from known bots and spiders" is enabled.

2. **Create a GA4 exploration** to isolate bot traffic:
   - Filter: device=desktop AND sessionMedium=(none) AND bounce rate > 90%
   - Check landing pages — bots often hit homepage or `/sitemap` repeatedly
   - Check screen resolution — bots often report 0x0 or unusual resolutions

3. **Consider adding bot detection headers** in `proxy.ts`:
   - Log `User-Agent` for desktop direct traffic with high bounce
   - Optionally block known scraper user-agents at the CDN/WAF level (CloudFront/SST)

4. **Create a "clean" GA4 audience** excluding suspected bot traffic for accurate reporting:
   - Audience: NOT (country in [China, Singapore, "not set"] AND medium = "(none)")

## Finding 3: Low CTR on Place Pages — Meta Titles Miss Search Intent

### Data

Top place pages by wasted impressions (CTR < 3%):

| Page | Impr | CTR | Pos | Top Query |
|------|------|-----|-----|-----------|
| `/` | 3,379 | 2.1% | 32.6 | "esdeveniments", "que fer aquest cap de setmana" |
| `/barcelona` | 1,093 | 2.4% | 53.5 | "actividades en barcelona" |
| `/el-masnou` | 732 | 2.9% | 7.7 | "agenda masnou", "agenda el masnou" |
| `/la-garriga` | 677 | 1.2% | 9.2 | "agenda la garriga" |
| `/vilassar-de-mar` | 546 | 1.8% | 15.8 | "agenda vilassar de mar" |
| `/malgrat-de-mar` | 517 | 2.9% | 9.8 | "agenda malgrat de mar" |
| `/maresme` | 492 | 2.4% | 30.6 | "agenda maresme" |
| `/canet-de-mar` | 492 | 1.8% | 7.2 | "canet de mar agenda" |

### Why This Happens (Codebase)

The meta title template in `messages/ca.json` for town pages is:
```
"metaTitle": "Què fer {label} - Agenda i plans {year}"
```

This generates titles like "Què fer al Masnou - Agenda i plans 2026". While semantically correct, it **doesn't match the exact query pattern** users search for. GSC shows users search "agenda [town]" or "agenda [town] [year]", not "Què fer a [town]".

The meta description template is:
```
"metaDescription": "Descobreix què fer {label} aquest {month}. Consulta concerts, festes, activitats familiars, tallers, exposicions i altres plans per gaudir del municipi."
```

This is generic — "gaudir del municipi" doesn't differentiate between towns or create urgency.

### 2026 SEO Trend Alignment

Google's AI Overviews (SGE) now dominate many event/activity queries, pulling from pages with the most directly relevant titles. The E-E-A-T emphasis means local specificity matters more than generic templates. Pages that explicitly mention the town name, current events/festivals, and temporal freshness signals will rank higher.

### Action Plan

1. **Update meta title template** in `messages/ca.json`:
   - Current: `"Què fer {label} - Agenda i plans {year}"`
   - Proposed: `"Agenda {label} {year} — Què fer avui i cap de setmana"`
   - This front-loads "Agenda [town]" to match the exact search query pattern

2. **Update meta description template** in `messages/ca.json`:
   - Current: generic "gaudir del municipi"
   - Proposed: `"Descobreix tots els esdeveniments {label}: concerts, fires, festes, mercats, exposicions i activitats familiars. Filtra per data i categoria. ✓ Actualitzat cada dia"`
   - Adds: specificity (town name via {label}), action verbs, freshness signal ("actualitzat cada dia"), trust mark (✓)

3. **Do the same for `es.json` and `en.json`** to improve Spanish/English page CTR

4. **Special case for homepage** (`root.metaTitle`):
   - Current: `"Què fer a Catalunya - Agenda cultural i plans {year}"`
   - Proposed: `"Esdeveniments.cat — Agenda d'activitats a Catalunya {year}"`
   - Reason: brand query "esdeveniments" at pos 18 needs brand name in title

5. **Special case for Barcelona** (`townNoDate.barcelona.metaTitle`):
   - Current: `"Què fer a Barcelona - Agenda i plans {year}"`
   - Proposed: `"Agenda Barcelona {year} — Esdeveniments, concerts i activitats avui"`

6. **Verify**: After deploying, monitor GSC CTR for these pages over 2-4 weeks. A well-optimized snippet should lift CTR from 2% to 5-8% at the same positions.

## Finding 4: Share Feature Nearly Dead — 14 Shares from 12,843 Users

### Data

| Event | Count | Users | Rate |
|-------|-------|-------|------|
| `share` | 14 | 12 | 0.09% of users |
| `outbound_click` | 434 | 332 | 2.6% of users |
| `view_event_page` | 456 | 327 | 2.5% of users |

For comparison, `outbound_click` (434 events) proves users engage with events — they just don't share them. The share rate (0.09%) is catastrophically low for an events discovery site where word-of-mouth should be the primary growth lever.

### Why This Happens (Codebase)

The share implementation in `app/e/[eventId]/components/EventShareBar.tsx` uses:
- **Mobile**: Native `navigator.share` API (via `NativeShareButton`) — only shows on mobile when API is supported
- **Desktop**: `CardShareButton` with dynamically loaded share icons (Twitter, Facebook, WhatsApp, etc.)

The `EventShareBar` is placed **inside the event detail page** (line 320 of `page.tsx`) — it's NOT in the card list view, so users only see share options after drilling into an event. Most discovery happens on list pages where there's no share affordance.

WhatsApp sharing IS implemented (`components/ui/common/staticShareButtons/index.tsx`, line 40), but it's buried inside the card share dropdown, not a primary CTA.

The `EventStickyCTA` (mobile sticky bar at bottom) has buttons for: external link, calendar, and favorite — but **NOT share**. This is a critical omission since mobile is where 82% of engaged sessions happen.

### 2026 SEO Trend Alignment

Social signals and referral traffic are increasingly important for E-E-A-T. WhatsApp sharing dominates in Spain/Catalonia for event discovery (it's how people coordinate weekend plans). The absence of easy WhatsApp sharing is a significant missed growth opportunity.

### Action Plan

1. **Add share to `EventStickyCTA`**: The mobile sticky CTA bar currently has 3 buttons (external link, calendar, favorite). Add a 4th share button that triggers `navigator.share` on mobile. This puts sharing at the user's fingertips after scrolling.

2. **Add share on event cards in list view**: Add a small share icon on each event card in the listing pages (not just detail pages). This lets users share events they discover without needing to open the detail page.

3. **Make WhatsApp the primary share option on desktop**: Instead of a generic share dropdown, show a prominent "Comparteix per WhatsApp" button since it's the dominant sharing channel for this audience.

4. **Track share method**: Verify the `share` GA4 event includes the `method` parameter (it does: `cardShareButton/index.tsx` sends `method: "whatsapp"`). Monitor which share method is most used.

## Finding 6: `/es/` and `/en/` Place Pages — Thin Content Bounce

### Data

| Page | Sessions | Bounce | Duration |
|------|----------|--------|----------|
| `/es/la-garriga` | 23 | **78%** | **5s** |

| `/barcelona/teatre` | 25 | **92%** | **10s** |

Users who land on non-Catalan place pages leave in 3-5 seconds, while the same page in Catalan has 0% bounce and 2+ minute sessions.

### Why This Happens (Codebase)

The i18n implementation uses `next-intl` with translations in `messages/{locale}.json`. All place pages render the same component (`app/[place]/page.tsx`) with translated UI strings. However:

1. **Event data is the same across languages** — events are stored in Catalan and not translated. So `/es/la-garriga` shows Catalan event titles with Spanish UI chrome.
2. **The meta description says "Descubre la agenda hoy..."** (Spanish) but the content is in Catalan — user expectation mismatch.
3. **No localized intro content** — there's no Spanish/English paragraph explaining the town or its events.

The `/barcelona/teatre` issue is different — it's a category filter page that likely shows zero or very few results, causing immediate abandonment.

### 2026 SEO Trend Alignment

Google's helpful content update specifically penalizes pages where the content doesn't match the language/intent suggested by the URL and meta. A Spanish URL serving Catalan content is a textbook example of unhelpful content.

### Action Plan

1. **Verify event rendering in non-Catalan locales**: Check if event titles/descriptions render in the user's language or always in Catalan. If always Catalan, consider:
   - Adding a small note: "Els esdeveniments es mostren en català" (in the user's language)
   - Or implementing event title translation (higher effort)

2. **For `/barcelona/teatre` (92% bounce)**: Check if this page has actual events. If it frequently shows empty results, add:
   - A "no events found" state with suggested alternatives
   - Redirect to `/barcelona` with a category filter suggestion
   - Consider noindexing empty category pages

3. **Add localized intro content** for top Spanish-searched towns (Barcelona, Tarragona):
   - Short paragraph in Spanish/English explaining what the page shows
   - This reduces cognitive dissonance when users land from a Spanish search

## Finding 7: Add-to-Calendar — Core Feature with 0.12% Usage

### Data

| Event | Count | Users | Rate |
|-------|-------|-------|------|
| `add_to_calendar` (GA4 custom event) | 25 | 16 | 0.12% of users |
| `add_to_calendar_view` (GA4) | 91 | 73 | 0.57% of users |

91 users opened the calendar dropdown (saw options) but only 25 actually added. That's a 27% completion rate — meaning the calendar UI works but not enough people discover it.

### Why This Happens (Codebase)

The `AddToCalendar` component (`components/ui/addToCalendar/AddToCalendar.tsx`) is rendered inside `EventCalendar` (`app/e/[eventId]/components/EventCalendar.tsx`, line 65), which appears in the event detail page's sidebar/info section.

The `EventStickyCTA` mobile bar (line 475 of `page.tsx`) DOES include a calendar icon that scrolls to the calendar section — so it IS accessible on mobile.

However, the calendar button requires:
1. User scrolls down to the event info section (or taps sticky CTA)
2. Clicks the calendar button to expand dropdown
3. Selects Google Calendar / Apple Calendar / iCal
4. Confirms in the external calendar app

That's 3-4 steps for a core action. For an events site, calendar add should be as frictionless as possible.

### 2026 SEO Trend Alignment

Google's Event rich results include a "Save" action that adds to Google Calendar directly from SERPs. If the site achieves Event rich results (Finding 12), this becomes less critical. But until then, on-site calendar add is the only conversion path.

### Action Plan

1. **Simplify the calendar flow**: Instead of a dropdown with 3 options (Google, Apple, iCal), detect the user's platform:
   - Mobile iOS → default to Apple Calendar
   - Mobile Android / Desktop → default to Google Calendar
   - Show the primary option as a button, secondary as "more options"

2. **Add calendar to EventStickyCTA as a direct action**: Currently the sticky CTA calendar button scrolls to the calendar section. Instead, make it trigger the calendar add directly (detect platform, open Google/Apple Calendar link).

3. **Track funnel**: The gap between `add_to_calendar_view` (91) and `add_to_calendar` (25) suggests 66 users abandoned. Track why — is the dropdown confusing? Are Google Calendar links broken?

## Finding 8: AI Referrer Tracking — 271 ChatGPT Sessions, 0 Events Tracked

### Data

| AI Source | GA4 Sessions (traffic source) | `ai_referrer` events |
|-----------|------------------------------|---------------------|
| chatgpt.com | 157 + 114 = **271** | **0** |
| perplexity | 16 | **0** |
| copilot.com | 8 | **0** |
| **Total AI traffic** | **295 (2% of total)** | **0** |

AI traffic is the site's **3rd largest traffic source** after Google organic and direct, yet the tracking event fires zero times.

### Why This Happens (Codebase)

The `ai_referrer` tracking in `app/GoogleScripts.tsx` (lines 290-321) checks `document.referrer` against a hardcoded list:

```javascript
const aiDomains = [
  'chat.openai.com',  // ← OLD domain
  'perplexity.ai',
  'gemini.google.com',
  'bard.google.com',  // ← Renamed to Gemini
  'claude.ai'
];
```

**The bug**: ChatGPT's current referrer domain is `chatgpt.com`, not `chat.openai.com`. OpenAI migrated to `chatgpt.com` in 2024. The tracking code uses the old domain, so it never matches.

Additionally, `perplexity.ai` should match but GA4 shows it as referrer source `perplexity` (without `.ai`) — the `document.referrer` may contain the full URL which should still match via `.includes()`. Needs investigation.

Also missing from the list: `copilot.com` (Microsoft Copilot), which sends 8 sessions.

### 2026 SEO Trend Alignment

AI-driven search is the #1 SEO trend for 2026. ChatGPT, Perplexity, and Copilot are increasingly replacing Google for discovery queries. Tracking AI referrals accurately is critical for understanding the channel mix shift and optimizing for AI visibility (AEO — Answer Engine Optimization). The site already has `/llms.txt` for AI discoverability, which is forward-thinking.

### Action Plan

1. **Fix the aiDomains list** in `app/GoogleScripts.tsx`:
   ```javascript
   const aiDomains = [
     'chatgpt.com',       // Current ChatGPT domain
     'chat.openai.com',   // Legacy ChatGPT domain
     'perplexity.ai',
     'gemini.google.com',
     'claude.ai',
     'copilot.microsoft.com',
     'copilot.com',
   ];
   ```

2. **Verify the tracking fires**: After deploying, visit the site from ChatGPT (ask ChatGPT to link to an event page), then check GA4 Realtime → Events for `ai_referrer`.

3. **Create a GA4 custom audience** for AI referral users to track their behavior separately (what pages they visit, engagement, conversions).

## Finding 12: No Event Rich Results Despite Complete Schema

### Data

GSC Search Appearance shows only:
- **FORUMS**: 3 impressions, 0 clicks
- **Event rich results**: NONE

### Why This Happens (Codebase)

The JSON-LD Event schema in `utils/schema-helpers.ts` is technically complete and well-implemented. It includes all required Google fields: name, startDate, endDate, location (Place + PostalAddress + GeoCoordinates), description, image, offers, organizer, eventAttendanceMode, eventStatus.

However, three issues may prevent rich results:

**Issue A: Dynamic `eventStatus` marks past events as "Completed"** (lines 246-259):
```typescript
if (endDateTime && now > endDateTime) {
  eventStatusValue = "https://schema.org/EventScheduled"; // wait, it sets EventCompleted
}
```
Most events in the index are past events (they remain indexed long after ending). Google doesn't show rich results for `EventCompleted` events.

**Issue B: `isAccessibleForFree` was explicitly removed** (line 308):
```typescript
// isAccessibleForFree removed: event.type defaults to "FREE" so this is unreliable.
```
Google uses this property for filtering in Event rich results.

**Issue C: All events have `price: 0`** even paid events (lines 224-242):
```typescript
logSchemaWarning(slug, "offers", "missing price for paid event");
return { ...baseOffer, price: 0, ... };
```
Setting price: 0 for paid events could be seen as misleading structured data.

### 2026 SEO Trend Alignment

Google's Event rich results are increasingly selective, favoring sites with high E-E-A-T and accurate structured data. Misleading price data (claiming free when not) can result in manual actions or trust penalties. On the positive side, Google expanded Event rich results to more languages and regions in 2025-2026, including Catalan.

### Action Plan

1. **Add `isAccessibleForFree`** only for confirmed free events:
   ```typescript
   ...(event.type === "FREE" && { isAccessibleForFree: true }),
   ```

2. **Fix paid event pricing**: For events where `event.type !== "FREE"`, omit the `price` field entirely rather than claiming 0. Google can infer "price varies" from the missing field.

3. **Test with Google's Rich Results Test**: Submit a LIVE future event URL (not past) at https://search.google.com/test/rich-results to verify Google can parse the schema.

4. **Request indexing** for 5-10 future event pages via GSC URL Inspection → "Request Indexing". This forces Google to re-evaluate the structured data.

5. **Check GSC Enhancements tab**: Navigate to GSC → Enhancements → "Event" to see if Google has detected any Event markup issues. If "Event" doesn't appear in the list, Google hasn't recognized the markup at all — investigate rendering issues.
